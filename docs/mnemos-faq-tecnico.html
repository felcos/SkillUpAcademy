Stack y Decisiones de Diseno

Por que C# / .NET?

Multiples razones convergentes: experiencia del autor (Felipe es desarrollador C# profesional), performance del runtimeNota Tecnica.NET 8 tiene un JIT compiler avanzado con tiered compilation, SIMD vectorization, y un garbage collector optimizado para baja latencia. En benchmarks TechEmpower, ASP.NET Core esta consistentemente entre los top 5 frameworks mas rapidos. Para un sistema que calcula CRR con 9 dimensiones por cada resultado de busqueda, el rendimiento importa., ecosistema maduro (EF Core, ASP.NET Core, SignalR), type safety fuerte que previene errores en tiempo de compilacion, y soporte LTS de Microsoft.

Por que SQL Server y no PostgreSQL?

SQL Server se eligio para desarrollo porque estaba disponible en el equipo de trabajo. Sin embargo, la migracion a PostgreSQLNota TecnicaPostgreSQL ofrece ventajas criticas para MNEMOS en produccion: pgvector con indices HNSW para busqueda vectorial O(log N) en lugar del brute force actual O(N), tsvector con indices GiST para full-text search nativo (actualmente se usa LIKE), jsonb para metadata con indices GIN, y es open source sin licencia comercial. La migracion es viable porque todo el SQL esta encapsulado en repositorios (capa Infrastructure). esta planeada para produccion, donde se aprovecharian pgvector (busqueda vectorial O(log N)), tsvector (full-text search nativo) y jsonb (metadata indexable).

Como escala MNEMOS?

Actualmente es single-server, pero la arquitectura permite escalar:

1

Read Replicas

Las busquedas (CRR, full-text, vectorial) pueden ir a replicas de lectura. Las escrituras van al primario.

2

Particionamiento

Los memors se pueden particionar por fecha. Los mas recientes (hot) en SSD rapido, los antiguos (cold) en storage economico.

3

pgvector + HNSW

En PostgreSQL, los indices HNSW permiten busqueda vectorial O(log N) en lugar del brute force actual O(N).

Cual es la complejidad algoritmica del CRR?

El CRR (Cognitive Retrieval Ranking)Funcionamiento InternoCRR v5 tiene 9 componentes ponderados: Semantic (0.22), RS (0.11), Evidence (0.11), Credence (0.07), DecisionOutcome (0.07), Context (0.14), Serendipity (0.09), Activation (0.09), Text (0.10). Cada componente se calcula para cada resultado candidato. Los pesos se calibraron durante los 303 debates PhD y se validaron con 10K usuarios simulados. tiene complejidad O(N) por la busqueda vectorial + O(K) por los 9 score components por resultado, donde K es constante. El cuello de botella es la busqueda vectorial. Con pgvector+HNSW: O(log N) + O(K) = O(log N).

Hay caching?

No todavia. El SAN (Spreading Activation Network)IntegracionEl SAN funciona como un pre-cache cognitivo: mantiene niveles de activacion in-memory para nodos (memors, schemas, beliefs). Cuando un memor se accede, activa nodos vecinos con propagation factor 0.5 y half-life de 300s. La proxima busqueda CRR usa estos niveles de activacion como factor alpha-8 (peso 0.09). Es cache "inteligente" — no cachea resultados sino que prioriza lo cognitivamente activo. funciona como pre-cache cognitivo in-memory. Redis esta planeado para produccion como cache de consultas frecuentes y session state.

Como se maneja la concurrencia?

EF Core optimistic concurrencyFuncionamiento InternoOptimistic concurrency funciona asi: cada fila tiene un RowVersion (timestamp). Cuando lees un registro, capturas su version. Cuando escribes, el UPDATE incluye WHERE RowVersion = @captured. Si otro proceso modifico la fila entre tu lectura y escritura, el WHERE no match, EF Core lanza DbUpdateConcurrencyException, y puedes reintentar. Esto evita locks pesimistas que degradan rendimiento. Nota: todavia no implementado en MNEMOS, planeado para produccion. via RowVersion esta planeado (no implementado aun). Actualmente, SQL Server transactions protegen las operaciones criticas (crear memor + append event + update embedding es una transaccion atomica).

Como se recupera el SAN despues de reinicio?

No se recupera — por disenoPrincipio de DisenoEl SAN es transiente a proposito. Al reiniciar, empieza en cero y se "recalienta" organicamente con el uso. La alternativa (persistir el estado) introduciria riesgo de state corruption y complejidad de sincronizacion. La perdida de activacion temporal es un precio aceptable por la garantia de consistencia. Bjork diria: la re-recuperacion tras el reinicio fortalece los nodos realmente importantes (desirable difficulty).. El SAN es transiente (in-memory only). Al reiniciar, los niveles de activacion empiezan en cero y se recalientan organicamente con el uso. Esto es intencional: evita state corruption y la re-recuperacion fortalece los nodos importantes (desirable difficulty aplicada al propio sistema).
