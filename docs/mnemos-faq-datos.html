Importacion de Datos

Puede importar desde Obsidian, Notion u otras herramientas?

Todavia no de forma automatica, pero esta planificado via la interfaz IDataSourceConnectorCodigo C#IDataSourceConnector es la abstraccion para fuentes de datos externas. Cada conector implementa esta interfaz para inyectar datos en MNEMOS.

public interface IDataSourceConnector
{
    string SourceName { get; }
    Task<IReadOnlyList<RawDataItem>>
        FetchNewItemsAsync(
            DateTime since,
            CancellationToken ct);
    Task<bool> TestConnectionAsync();
}
. Obsidian seria el mas facil de integrar porque ya usa archivos .md — que es exactamente el formato nativo de MNEMOS para contenido.

Para Notion se necesitaria un conector que use la API de Notion para extraer paginas y convertirlas a memors. Para Roam/Logseq, similar con sus respectivos formatos de exportacion.

Puede captar datos de navegacion web?

Futuro. Se planea un conector IDataSourceConnector para el navegadorIntegracionEl conector de navegador funcionaria como una extension que captura: paginas guardadas como bookmarks, selecciones de texto, y opcionalmente historial de navegacion. Cada item se convierte en un memor con SourceContext que incluye URL, titulo, y timestamp. El Guardian filtraria contenido no relevante. que podria capturar bookmarks, selecciones de texto y opcionalmente historial. Cada item capturado se convertiria en un memor con su SourceContext (URL, titulo, fecha).

Puede captar datos del IDE?

Futuro. Se planea un conector para VS Code y JetBrainsNota TecnicaEl conector de IDE captaria: snippets guardados, decisiones tecnicas (commits con mensajes especificos), errores resueltos, y patrones de codigo frecuentes. Esto alimentaria especialmente al motor Kahneman (decisiones tecnicas) y schemas de Piaget (conocimiento tecnico que evoluciona).. La integracion mas inmediata es via MCP Server, que ya permite a Claude Code leer y crear memors directamente desde el terminal.

Hay bulk import?

No hay una herramienta dedicada todavia. Sin embargo, la API RESTFlujo de DatosPOST /api/memors acepta un memor individual. Para bulk import, se puede hacer un script que itere sobre archivos y haga POST secuenciales. Cada POST pasa por el pipeline completo: Guardian validation, ContentHash, embedding, SQL Server persist, DomainEvent append. Rate limit del Guardian: 100 memors/hora por source_agent. permite crear memors programaticamente. Un script en cualquier lenguaje puede iterar sobre archivos y hacer POSTs secuenciales a POST /api/memors. Cada memor pasa por el pipeline completo de validacion y procesamiento.

1. Recepcion
La peticion llega al MemorsControllerCodigo C#El controller es delgado — solo valida el DTO y delega al servicio.
[HttpPost]
public async Task<IActionResult> CreateAsync(
    CreateMemorRequest request)

 via POST. El DTO incluye contenido, tipo, dominios y contexto opcional.
2. Guardian
GuardianAgentFuncionamiento InternoEl Guardian valida: formato correcto, contenido no vacio, rate limiting (100/hora por source_agent), sanitizacion de prompt injection (detecta patrones como "ignore previous instructions"), inmutabilidad de campos protegidos (SS nunca decrece en updates), y source_agent no nulo para agentes. valida formato, rate limiting, prompt injection sanitization y campos protegidos.
3. ContentHash
Se genera un hash SHA-256 del contenido para detectar duplicados exactos y rastrear modificaciones.
4. Embedding
El IAIEngineIntegracionIAIEngine es la abstraccion para TODAS las operaciones de AI. Dos implementaciones: OpenAiCompatibleEmbeddingEngine (usa API OpenAI o compatible como Together AI, Ollama) y StubAIEngine (embeddings locales bag-of-words). La interfaz tiene un solo metodo para embeddings: GenerateEmbeddingAsync(string text). El embedding resultante (float[]) se almacena junto al memor. genera un vector de embedding (float[]) del contenido. ~200ms local, ~500ms con OpenAI.
5. Persistencia
El memor se guarda en SQL Server con SS=1.0 (Storage Strength inicial), RS=1.0 (Retrieval Strength inicial), timestamp, y el embedding.
6. Event Sourcing
Se crea un DomainEventFlujo de DatosCada mutacion de estado genera un DomainEvent en la tabla DomainEvents con: AggregateId (el memor), EventType ("MemorCreated"), Payload (JSON con el estado completo), Actor (usuario o agente), y timestamp. Esto permite reconstruir el historial completo de cualquier entidad — es el fundamento del event sourcing. de tipo "MemorCreated" con el estado completo en JSON.
7. Bus Cognitivo
El evento se publica en el CognitiveEventBusFuncionamiento InternoEl Bus Cognitivo tiene 11 tipos de evento. Cuando se crea un memor, se emite NewMemorEncoded. Los suscriptores incluyen: SchemaService (intenta asimilar el memor en schemas existentes), SpreadingActivationNetwork (activa nodos relacionados), y Discoverer (busca conexiones no obvias). Damping: 0.3 por salto, maximo 5 saltos.. Los motores suscritos reaccionan: Piaget intenta asimilar en schemas, SAN activa nodos relacionados.

Que pasa si la API de OpenAI no esta disponible?

MNEMOS aplica el Principio 5: Graceful DegradationPrincipio FilosoficoEl Principio 5 establece que MNEMOS debe funcionar de forma util incluso sin conexion a APIs externas, sin biometricos, sin todos los motores activos. Cada capa se degrada con gracia — pierde precision pero no funcionalidad. El StubAIEngine es la materializacion de este principio para embeddings.«Un sistema que solo funciona en condiciones perfectas no funciona.». El StubAIEngine toma el control automaticamente y genera embeddings locales usando bag-of-words + feature hashing. La busqueda funciona (por keywords), pero pierde la comprension semantica profunda que ofrece OpenAI.

Edicion y Borrado

Puedo editar un memor despues de crearlo?

Si, via PUT /api/memors/{id} o el tool MCP create_memor (con update). El Guardian valida los cambiosFuncionamiento InternoCuando se edita un memor, el Guardian verifica: (1) que SS no decrece (invariante fundamental de Bjork), (2) que source_agent esta presente si es un agente quien edita, (3) que no hay prompt injection en el nuevo contenido, (4) rate limiting. El ContentHash se recalcula y el embedding se regenera con el contenido nuevo.: SS nunca puede decrecer (invariante Bjork), el embedding se regenera, y un DomainEvent "MemorUpdated" registra el cambio completo.

Puedo borrar un memor?

MNEMOS no borra — archivaCiencia CognitivaBjork & Bjork (1992) demostraron que la informacion almacenada en memoria a largo plazo nunca se "borra" realmente — solo se vuelve inaccesible (RS baja a ~0). MNEMOS implementa esto literalmente: un memor archivado tiene ArchivalStatus cambiado, pero su SS permanece intacta. El event sourcing mantiene el historial completo.Bjork, R.A. & Bjork, E.L. (1992). A New Theory of Disuse.. Cuando "borras" un memor, su ArchivalStatus cambia de Active a Archived. El memor deja de aparecer en busquedas, pero:

  • Su Storage Strength (SS) permanece intacta
  • El event sourcing conserva todo el historial
  • Se puede des-archivar en cualquier momento
  • Los links existentes se marcan como inactivos pero no se eliminan

Por que no borrado real? Porque SS nunca decrece (invariante Bjork). Borrar un memor violaria este principio. Ademas, el historial de eventos es sagrado para event sourcing — permite reconstruir el estado en cualquier punto del pasado (Cognitive Time Travel futuro).

Que pasa si meto datos basura?

El GuardianNota TecnicaEl GuardianAgent actua como portero: valida formato (contenido no vacio, tipo valido), aplica rate limiting (100/hora por source_agent), detecta prompt injection, y verifica inmutabilidad de campos protegidos. Sin embargo, no juzga la calidad del contenido — eso es responsabilidad del usuario. Si escribes "asdfgh", el Guardian lo dejara pasar si tiene formato valido. valida formato y rate limiting, pero no juzga calidad de contenido. Si escribes algo sin sentido, entrara al sistema. La calidad del contenido es responsabilidad del usuario. En futuro, el agente Librarian podria sugerir mejoras.

Clasificacion y Conexiones

Tengo que clasificar manualmente cada nota?

No obligatoriamente. Cada memor tiene un MemorType (Episodic, Semantic, Procedural, etc.) que actualmente se especifica al crear. En futuro, se planea auto-clasificacionFuncionamiento InternoLa auto-clasificacion usaria el IAIEngine para analizar el contenido y sugerir el tipo mas probable. Un memor que describe "ayer tuve una reunion con..." seria Episodic. Uno que explica "la ley de Ohm establece..." seria Semantic. Uno con pasos "para deployar, primero..." seria Procedural. El usuario siempre puede corregir la clasificacion. basada en analisis semantico del contenido.

Como se vincula un memor a un schema?

Automaticamente. Cuando se crea un memor, el Bus Cognitivo emite NewMemorEncoded. El motor Piaget, suscrito a este evento, ejecuta SchemaService.TryAssimilateAsync()Codigo C#El servicio intenta encajar el memor en schemas existentes por similaridad semantica.

public async Task TryAssimilateAsync(
    Guid memorId,
    CancellationToken ct)
{
    // 1. Obtener embedding del memor
    // 2. Buscar schemas con mayor
    //    similaridad coseno
    // 3. Si similaridad > umbral:
    //    asimilar (anadir a schema)
    // 4. Si no encaja en ninguno:
    //    puede triggear acomodacion
    //    (modificar schema existente)
    //    o crear schema nascent
}
que compara el embedding del memor con los schemas existentes. Si la similaridad supera el umbral, el memor se asimila al schema. Si no encaja en ninguno, puede triggerear una acomodacion (modificar el schema) o quedar sin clasificar hasta que surja un patron.

Como creo una creencia (belief)?

Actualmente via API REST (POST /api/beliefs) o MCP tool (create_belief). Cada belief tiene: nombre, descripcion, credence_ideal y credence_actualCiencia CognitivaEl dual-track Bayesian es una innovacion de MNEMOS: credence_ideal es lo que la evidencia dice que deberias creer (calculado objetivamente). credence_actual es lo que realmente crees (subjetivo). La diferencia entre ambos es el bias_gap — una medida cuantificada de sesgo cognitivo que el motor detecta automaticamente.Inspirado en epistemologia Bayesiana formal. El dual-track fue propuesto durante los 303 debates PhD. (ambos entre 0 y 1), dominio y evidencia asociada. En futuro, el sistema podria auto-inferir beliefs desde patrones en los memors.

Como registro una decision?

Via POST /api/decisions o MCP tool create_decision. Cada decision incluye: descripcion, predicted_outcomeFuncionamiento InternoEl motor Kahneman registra la prediccion del usuario ANTES del resultado real. Cuando el resultado llega (via resolve_decision), compara prediccion vs realidad. Esto alimenta success_rate del DecisionPattern asociado. Con el tiempo, MNEMOS puede mostrar: "En decisiones de este tipo, predices correctamente el 73% de las veces" o "Tiendes a ser sobre-optimista en estimaciones de tiempo (System 1)"., contexto (memor asociado), schema, beliefs relevantes, y sistema de razonamiento (System1 o System2). Cuando el resultado llega, se resuelve con resolve_decision.

Los schemas se crean automaticamente?

Actualmente son manuales. El agente DiscovererIntegracionEl DiscovererAgent busca conexiones no obvias entre memors. Usa el IAIEngine para encontrar memors semanticamente relacionados que no estan vinculados. Cuando encuentra un cluster de memors no conectados pero tematicamete coherentes, sugiere la creacion de un schema. Implementado en Phase 1 Sprint 5 pero la auto-creacion de schemas esta en el roadmap. detecta clusters de memors no conectados pero tematicamente coherentes, y en futuro sugerira la creacion automatica de schemas. El ciclo de vida de un schema sigue a Piaget: nascent → developing → stable → rigid → obsolete.
